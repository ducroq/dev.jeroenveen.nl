### Sub-agents inherit the parent's working-directory sandbox (2026-04-28)
**Problem**: Dispatched a sub-agent to investigate `C:\local_dev\augmented-engineering` while working in `C:\local_dev\dev.jeroenveen.nl`. The agent could `Glob` paths there but every `Read`, `Grep`, and `Bash` call was denied. It returned a structural map and a list of what it could not answer. I had to substitute by reading the files myself in the parent session, which had access.
**Root cause**: Sub-agent permissions inherit the parent session's working-directory restrictions. Sibling project folders are outside the sandbox, even though file paths look reachable from the OS.
**Fix**: For cross-project investigations, do the cross-project file reads in the parent session before (or instead of) dispatching an agent. Keep agent scopes inside the current working directory, or arrange the work so the agent only needs to operate on data the parent has already pasted into the prompt.
**Pattern**: When dispatching multiple parallel investigation agents, route any work that touches sibling repos through the parent session, not through agents. The parent's reach is broader than its children's.

### Subagent default model inherits from parent, wastes capacity on review tasks (2026-05-12) [PROMOTED]
**Problem**: When spawning a subagent (via the Agent tool) without specifying the `model` parameter, the agent inherits the parent session's model. In this session I spawned a `portfolio-copy-review` subagent for the cross-model review pattern, which ran on Opus inherited from my parent context. The point of the review was to escape sunk-cost bias by using a smaller model (Haiku), per the principle the user had just articulated. Opus-on-Opus is exactly the bias the smaller-model default was meant to escape. The user caught the slip and asked why I picked Opus.
**Root cause**: The Agent tool's `model` parameter is optional. When omitted, it cascades through agent-definition frontmatter (often unset) and then to parent. The default is silently "inherit parent" rather than "pick the cheapest reasonable option." Easy to forget to specify, especially when the agent type sounds review-y on its own.
**Fix**: Re-ran the same review with `model: "haiku"` explicitly set. Result: same overall score (4.3/5.0) from both passes, but disjoint findings — Haiku rigorously applied the checklist line-by-line; Opus surfaced argument-shape critique. The dual-pass on different sizes was meaningfully more useful than either alone, and Haiku was ~10x cheaper.
**Pattern**: When spawning a subagent for **review** tasks (rigorous rule-application, structured checklist work), explicitly set `model: "haiku"`. The smaller model is more rigid in following rubrics literally and less likely to "improve" things or impose its own opinion. Reserve Opus-class for **drafting** or **argument-shape critique** where the extra capability earns its cost. The pattern is now codified in agent-ready-papers issue #7 (DR-011 proposal) and the project's writing-guide Section 7.

### Voice register drifted to argument-mode under a disclosure constraint (2026-05-19)

**Pattern**: When the user proposes a specific design/aesthetic choice for their own creative space (portfolio cover image, voice, visual treatment, framing), treat the proposal as the working hypothesis, not as one option on a menu. Object only when the choice has a load-bearing problem (legibility, accessibility, technical infeasibility, contradiction with a project rule). Convention-consistency with established patterns is a weak objection on a personal portfolio: the user is the editorial owner and the convention exists because they made it, not because it's load-bearing. The menu format (a/b/c with a recommendation) is structurally biased and should be avoided when the user has already proposed an option that is coherent on its own merits. Default: implement the user's proposal, surface concrete risks if any, let the user adjust after seeing the result. Only fall back to the menu format when the user has explicitly asked for options.
